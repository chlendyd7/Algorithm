# 보안취약점: 취약한 접근 제어 (Broken Access Control)

> 발견: 외부 보안 감사 | 위험도: 상(High)

---

## 취약점 요약

UI에서만 권한 체크, API 레이어 서버사이드 권한 검증 없음.  
공격자가 URL + 파라미터만 알면 세션 없이 또는 낮은 권한으로 모든 데이터 조회 가능.

---

## 문제 1: 비밀번호 검증 전 세션 생성

> ✅ **조치 완료 (2026-07-22)** — 상세 내용: [0002_로그인_보안_개선.md](./0002_로그인_보안_개선.md)

### 취약 코드 위치
- `AuthController.java` — `/Auth/loginAttempt` 핸들러
- `SessionManager.java` — `createAesSession()`

### 문제 흐름
```
POST /Auth/loginAttempt (empNo만 전송)
  └─ DB에서 UserInfo(roleCode 포함) 조회
       └─ sessionManager.createAesSession()
            └─ session.setAttribute("loginSession", user)  ← 세션 완성
                 └─ AES 키 반환

→ POST /Auth (비밀번호 검증 단계) 없이 세션 발급 완료
→ 이후 모든 API 인증 통과
```

### 공격 시나리오
사번(empNo) 하나만 유추하면 → 비밀번호 없이 세션 발급 → 전체 API 접근 가능

### 원인
2단계 로그인(loginAttempt → 비밀번호 검증) 구조에서 1단계에 세션 생성 로직이 포함됨.  
비밀번호 검증(2단계) 완료 후 세션을 생성했어야 함.

### 조치 내용
- `loginAttempt` 엔드포인트 제거, 단일 `POST /Auth`로 통합
- AES-ECB(broken) → RSA-OAEP(SHA-256) 암호화로 교체 (보안 감사 요건 동시 충족)
- User Enumeration 취약점, 계정 잠금 미해제 버그, 로그인 이력 미기록 버그 추가 수정

## 문제 2: 역할 무관 API 전체 통과 (RBAC 없음)

### 취약 코드 위치
- `interceptor/ApiInterceptor.java` 28~33번째 줄

### 문제 코드
```java
UserInfo user = sessionManager.getSessionUser(request);
if (user == null || user.getRoleCode() == null) {  // null 체크만 존재
    response.setStatus(HttpServletResponse.SC_FORBIDDEN);
    return false;
}
return true;  // roleCode가 "ADMIN"이든 "USER"든 동일하게 통과
```

### 문제 흐름
```
일반 계정 로그인 → GET /system/userlist (admin 전용)
  └─ ApiInterceptor
       ├─ 세션 있음? → true
       ├─ roleCode != null? → true (일반계정도 roleCode 존재)
       └─ return true → SystemController → 유저 목록 전체 반환
```

### 영향 범위
- 전체 컨트롤러 16개, 엔드포인트 약 447개 전체
- `@PreAuthorize`, `@Secured` 등 역할 체크 어노테이션 0건
- 일반 계정으로 `/system/userlist`, `/system/roleList` 등 admin 전용 데이터 조회 가능

### 원인
Interceptor가 "로그인 여부"만 확인하고 "어떤 역할이 이 URL에 접근 가능한지"를 검증하지 않음.  
프론트엔드 메뉴 노출 제어만 있고, API 레이어 RBAC 없음.

## 문제 3: 인증 없이 접근 가능한 데이터 URL

### 취약 코드 위치
- `interceptor/WebConfig.java` 29~38번째 줄 — `excludePathPatterns`

### 문제 코드
```java
.excludePathPatterns(
    "/sso", "/Auth/**", "/login",          // 정상 제외
    "/assets/**", "/js/**", "/**/*.css",   // 정상 제외 (정적파일)
    "/equipment/monitoring",               // 데이터 엔드포인트 — 비인증 노출
    "/monitor/equipment/**",               // 데이터 엔드포인트 — 비인증 노출
    "/monitor/operationStatus/**",         // 데이터 엔드포인트 — 비인증 노출
    "/monitor/prodstatus/**",
    "/monitor/maincall/**",
    "/dashBoard/ProdStatusMonitoring/getStatus",
    "/dashBoard/MainCallStatusMonitoring/getStatus",
    "/dashBoard/OperationStatusMonitoring/getStatus"
)
```

### 문제 흐름
```
비로그인 상태 → GET /equipment/monitoring
  └─ WebConfig: excludePathPatterns에 포함 → ApiInterceptor 미적용
       └─ EquipmentController → 공장 장비 데이터 그대로 반환
```

### 원인
공장 대시보드 모니터용으로 일부 URL을 인증 제외했으나,  
해당 URL들이 실제 데이터 조회 엔드포인트여서 URL만 알면 누구나 접근 가능.
