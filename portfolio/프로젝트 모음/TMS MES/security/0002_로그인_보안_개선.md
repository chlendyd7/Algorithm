# 로그인 보안 개선 작업 기록

> 작업 일자: 2026-07-22  
> 시스템: TMS MES (통합 열관리 시스템)  
> 목적: 보안 감사 요건 충족 + 레거시 코드 정리

---

## 배경

보안 감사 체크리스트 항목 중 **"중요 정보 추가 암호화 (AES 등)"** 요건이 있었음.  
기존 로그인 흐름을 검토하다가 구조적 문제를 발견하고 전면 개선하기로 결정.

---

## 기존 구조 분석

### 로그인 2단계 흐름

```
[1] POST /Auth/loginAttempt
    - 사용자 존재 확인
    - AES 키 생성 → 세션 저장
    - AES 키 클라이언트에 반환

[2] POST /Auth
    - 클라이언트: SHA512(pw) → AES_ECB(SHA512(pw), 서버키)
    - 서버: AES 복호화 → SHA512 해시 비교
```

### 문제점 발견

**1. AES-ECB 모드 사용 (치명적 결함)**

```js
// 기존 코드
cryptoJs.AES.encrypt(encryptedPwd, key, {
    mode: cryptoJs.mode.ECB,  // ← 문제
    padding: cryptoJs.pad.Pkcs7,
})
```

- ECB 모드는 동일 입력 → 동일 출력 (IV 없음)
- SHA512(password)는 고정값 → AES-ECB 결과도 항상 동일
- **Replay Attack 방지 의도였으나 실제로는 효과 없음**
- 즉, 보안 수준이 SHA512만 사용하는 것과 동일

**2. User Enumeration 취약점**

```
/loginAttempt → 사용자 없음: 401 "사용자를 찾을 수 없습니다"
/loginAttempt → 사용자 있음: 200 + AES 키 반환
```

공격자가 이 차이로 유효한 사번 목록 수집 가능 (계정 존재 여부 노출)

**3. 로그인 성공 시 잠금 미해제 버그**

```java
// loginAttemptSuccess SQL - 기존 상태
<update id="loginAttemptSuccess">
--  UPDATE ma_user SET login_try=0, is_lock='N', lock_dt=null
--  WHERE emp_no = #{empno};  ← 주석 처리된 채로 방치
</update>
```

10회 실패로 잠긴 계정이 올바른 비밀번호로 로그인해도 영구 잠금 상태 유지

**4. 로그인 성공 이력 미기록 버그**

```java
// 기존 코드
if (user.passwordCheck(decryptedPassword)) {
    // ...
    return "로그인 성공"; // ← 여기서 return
}
loginhistortService.putLoginHistory(userlog); // ← 성공 시 도달 불가
```

성공 케이스는 로그인 이력에 기록되지 않았음

## 고민했던 내용

### 1단계 통합이 가능한가?

> "loginAttempt를 그냥 Login 하나로 처리 못함?"

**검토 사항:**
- HTTPS 사용 중이면 TLS가 이미 전송 구간 암호화
- AES-ECB가 broken이므로 현재 구현은 SHA512만 쓰는 것과 보안 수준 동일
- 따라서 단순화해도 보안 수준 변화 없음

**결론:** HTTPS 환경에서는 단순화 가능, 오히려 코드 복잡도만 줄임

---

### 보안 감사 요건 충족을 위한 암호화 방식 비교

> 요건: "중요 정보 추가 암호화 (AES 등)"

#### Option A: AES-CBC + loginAttempt 복구

```
loginAttempt → 서버: AES키 + IV 반환
클라이언트: AES-CBC(SHA512(pw), 키, IV)
POST /Auth: 암호화된 pw 전송
```

- 기존 구조 일부 유지 (2단계)
- AES-ECB → AES-CBC로 교체 필요
- 키가 네트워크를 타는 것이 근본적 약점

#### Option B: RSA 공개키 암호화

```
서버: RSA 공개키 노출 (GET /Auth/publicKey)
클라이언트: RSA_OAEP(SHA512(pw), 공개키)
POST /Auth: 암호화된 pw 전송
서버: 개인키로 복호화
```

**비유로 설명:**
- A방식: 서버가 자물쇠 + 열쇠 같이 보냄 → 중간에 탈취 시 열쇠도 탈취
- B방식: 서버가 자물쇠만 보냄, 열쇠는 서버 밖으로 절대 안 나감

| 항목 | A (AES-CBC) | B (RSA) |
|---|---|---|
| 열쇠 이동 | 서버→클라이언트 전송 | 서버 밖 이동 없음 |
| 탈취 시 | 열쇠+자물쇠 동시 탈취 | 자물쇠만 탈취, 복호화 불가 |
| loginAttempt 필요 | 필요 | 불필요 (단일 호출) |
| 코드 수정량 | 많음 | 적음 |
| 보안 강도 | 중간 | 높음 |

**RSA 키 지속성 고민:**
> "서버 껐다 키면 RSA 키쌍 매번 새로 만들어지는 거임?"

- 서버 시작 시 매번 새 키쌍 생성 → **문제 없음**
- 로그인 시 공개키를 매번 새로 fetch하면 재시작과 무관
- 영속성 필요 시 파일/DB 저장 옵션 있지만 내부망 시스템에는 과함

**선택: Option B (RSA)**

## 구현 내용

### 백엔드 변경 (AuthController.java)

**RSA 키쌍 생성 (서버 시작 시 1회)**

```java
private static final KeyPair RSA_KEY_PAIR;
static {
    try {
        KeyPairGenerator keyGen = KeyPairGenerator.getInstance("RSA");
        keyGen.initialize(2048);
        RSA_KEY_PAIR = keyGen.generateKeyPair();
    } catch (Exception e) {
        throw new RuntimeException("RSA 키 생성 실패", e);
    }
}
```

**공개키 엔드포인트**

```java
@GetMapping("/publicKey")
public String getPublicKey() {
    return Base64.getEncoder().encodeToString(RSA_KEY_PAIR.getPublic().getEncoded());
}
```

**RSA 복호화 메서드 (핵심 포인트)**

```java
private static String rsaDecrypt(String encryptedBase64) throws Exception {
    Cipher cipher = Cipher.getInstance("RSA/ECB/OAEPPadding");
    OAEPParameterSpec params = new OAEPParameterSpec(
        "SHA-256", "MGF1", new MGF1ParameterSpec("SHA-256"), PSource.PSpecified.DEFAULT
    );
    cipher.init(Cipher.DECRYPT_MODE, RSA_KEY_PAIR.getPrivate(), params);
    byte[] decrypted = cipher.doFinal(Base64.getDecoder().decode(encryptedBase64));
    return new String(decrypted);
}
```

**단일 로그인 흐름 (POST /Auth)**

```java
String decryptedPassword = rsaDecrypt(userSearch.getPassword());
userSearch.setPassword(decryptedPassword);
UserInfo user = authService.Login(userSearch);
// ... 이후 passwordCheck, 세션 생성
```

### 프론트엔드 변경 (LoginView.vue, MobileLoginView.vue)

```js
const login = async () => {
    // 1. SHA512 해시
    const sha512Pwd = cryptoJs.SHA512(password.value).toString().normalize().trim();

    // 2. 서버 공개키 fetch
    const keyRes = await axios.get(pathPrefix + '/Auth/publicKey', { withCredentials: true });
    const keyData = Uint8Array.from(atob(keyRes.data), c => c.charCodeAt(0));

    // 3. 브라우저 내장 SubtleCrypto로 RSA 암호화 (외부 라이브러리 불필요)
    const publicKey = await window.crypto.subtle.importKey(
        'spki', keyData.buffer,
        { name: 'RSA-OAEP', hash: 'SHA-256' },
        false, ['encrypt']
    );
    const encryptedBuffer = await window.crypto.subtle.encrypt(
        { name: 'RSA-OAEP' }, publicKey,
        new TextEncoder().encode(sha512Pwd)
    );
    const encryptedPwd = btoa(String.fromCharCode(...new Uint8Array(encryptedBuffer)));

    // 4. 단일 POST
    await axios.post(pathPrefix + '/Auth', { empno: empno.value, password: encryptedPwd }, { withCredentials: true });
    router.push('/');
}
```

## 트러블슈팅

### 이슈 1: MGF1 해시 알고리즘 불일치

**증상:** RSA 복호화 예외 발생 → 500 반환 → 프론트가 성공으로 오인 → 401

**원인:**

| 항목 | Java JCE `OAEPWithSHA-256AndMGF1Padding` | SubtleCrypto `{ hash: 'SHA-256' }` |
|---|---|---|
| OAEP 해시 | SHA-256 | SHA-256 |
| MGF1 해시 | **SHA-1** (JCE 기본값) | **SHA-256** (Web Crypto 스펙 강제) |

Java JCE의 `OAEPWithSHA-256AndMGF1Padding` 문자열은 OAEP 해시만 지정하고 MGF1 해시는 SHA-1로 기본 처리됨. 반면 Web Crypto API는 지정한 hash를 MGF1에도 동일하게 적용.

**해결:** `OAEPParameterSpec`으로 MGF1 해시를 명시적으로 SHA-256 지정

```java
// 잘못된 방법
Cipher.getInstance("RSA/ECB/OAEPWithSHA-256AndMGF1Padding");
// MGF1 해시가 SHA-1로 처리됨

// 올바른 방법
Cipher.getInstance("RSA/ECB/OAEPPadding");
new OAEPParameterSpec("SHA-256", "MGF1", new MGF1ParameterSpec("SHA-256"), PSource.PSpecified.DEFAULT);
// OAEP, MGF1 모두 SHA-256 명시
```

### 이슈 2: catch 블록 HTTP 상태코드 미설정

**증상:** 복호화 실패 시 200 반환 → 프론트가 성공으로 판단 → router.push('/') → GET /Auth/User 401

**원인:** catch 블록에서 `response.setStatus()` 누락

**해결:** catch에서 500 명시

```java
} catch (Exception e) {
    logger.error("Login error: {}", e.getMessage(), e);
    response.setStatus(HttpServletResponse.SC_INTERNAL_SERVER_ERROR);
    return e.getClass().getSimpleName() + ": " + e.getMessage();
}
```

## 배운 점 (시니어 개발자 관점)

1. **의도와 구현 사이의 간극 확인**: 보안 코드는 "적용했다"가 아니라 "실제로 효과가 있는가"를 검증해야 함. AES-ECB는 개발자가 보안을 의도했지만 실제로는 효과 없는 코드였음.

2. **라이브러리 기본값 신뢰 금지**: Java JCE `OAEPWithSHA-256AndMGF1Padding`처럼 이름만 보고 동작을 가정하면 안 됨. 스펙 문서로 기본값 확인 필수.

3. **에러 처리의 중요성**: catch 블록에서 HTTP status 미설정 → 클라이언트가 잘못된 판단 → 디버깅 난이도 급상승. 에러는 명확하게 전달해야 함.

4. **단순화가 곧 보안**: 불필요한 복잡도(AES-ECB 2단계)를 제거하고 더 안전한 단순 구조(RSA 1단계)로 교체. 코드가 단순할수록 취약점도 줄어듦.

5. **암호화 표준 간 호환성**: 서로 다른 플랫폼(Java JCE ↔ Web Crypto API) 간 암호화 사용 시 알고리즘 파라미터를 양쪽 스펙 문서 기준으로 명시적으로 맞춰야 함. 이름이 같아도 기본값이 다를 수 있음.
