주요 성과 — 원인 / 분석 / 해결
1
Tool-Calling 기반 자기검증 Agentic Loop 구현
원인

단발성 1회 프롬프트 호출 구조에서 AI가 프로젝트의 현재 API 경로·라우팅 상태를 모른 채 코드를 생성해, 이미 존재하는 경로와 충돌하거나 존재하지 않는 import를 참조하는 오류가 반복 발생

분석

생성 시점의 런타임 상태를 AI에게 주입하지 않는 것이 근본 원인. Claude API의 tool_use 기능으로 AI가 생성 도중 직접 정보를 조회하도록 구조를 바꿔야 함

해결

tool_use → tool_result → 재호출 → end_turn 루프를 Spring Boot에서 직접 구현. AI가 생성 중 list_vibe_apis 도구를 호출해 현재 API 목록을 조회한 뒤 충돌 없는 코드를 생성하는 자기검증 흐름 완성

2
기업 AI API 제약 대응 — HChat Fallback 이중 설계
원인

사내 AI 시스템(HChat)이 tool_use를 지원하는지 공식 문서에 명시되지 않아, Agentic Loop를 적용할 수 있는지 불확실한 상황에서 설계를 진행해야 했음

분석

실제 HChat API 호출 후 응답 로그를 직접 파싱해 round=0 tool_use → list_vibe_apis 호출 → round=1 end_turn 흐름을 확인. tool_use 지원이 검증됨. 동시에 미지원 케이스 대응 설계 필요성 인지

해결

tool_use 지원 시 → Agentic Loop 실행, 미지원 또는 오류 시 → 기존 정적 프롬프트 생성 방식으로 자동 전환하는 fallback 이중 설계 구현. PoC 단계에서도 안정적 실행 보장

3
OpenCode 소스 분석 → 자체 Agent 아키텍처 개선안 도출
원인

반복 편집 시 AI가 이전 대화 내용을 기억하지 못하는 context loss, 단일 AIService(685줄)가 모든 AI 역할을 담당해 역할별 모델 최적화 불가

분석

production AI 에이전트 OpenCode의 핵심 파일(compaction.ts, overflow.ts, agent.ts, system.ts)을 직접 파싱. 6개 Agent 역할 분리 구조, Compaction(누적 요약) 전략, 역할별 모델 자동 선택 메커니즘 확인

해결

VibeCoding 개선안 설계 완료: ① context loss → Haiku 기반 대화 요약을 다음 요청에 주입, ② AIService → CodeGeneratorAgent / EditAgent / SummaryAgent 3단계 분리, 역할별 모델 최적화로 비용 절감 구조 도출

4
MCP 서버 구현 — 운영자용 도구 계층 표준화
원인

배포 버전 관리·롤백·상태 확인 등 운영 작업을 수동으로 처리해야 했고, Claude Desktop 같은 외부 AI 클라이언트가 VibeCoding 인프라를 직접 제어할 수 있는 표준 인터페이스가 없었음

분석

MCP(Model Context Protocol)를 통해 모든 AI 클라이언트가 SSE 방식으로 외부 서비스 도구를 호출할 수 있음을 확인. 운영/배포 도구를 MCP 레이어로 표준화하면 어떤 클라이언트든 연결 가능

해결

FastAPI 기반 MCP SSE 서버 구현 (8개 도구): 배포 버전 목록 조회·롤백·상태 확인·신규 버전 커밋 등. Claude Desktop에서 자연어로 "v2로 롤백해줘" 호출 → FastAPI → Git checkout → Vite 재빌드 자동 실행

왜 이 경험이 강한가 — 경쟁자 대비
관점 LangChain 튜토리얼 기반 프로젝트 Vibe Coding Studio
LLM 호출 구조 프레임워크 추상화에 의존, 내부 동작 미이해 tool_use → tool_result → 재호출 루프를 직접 구현해 내부 메커니즘 완전 이해
기업 환경 경험 OpenAI / Anthropic 공식 API만 사용 사내 AI API(HChat) tool_use 지원 여부를 로그로 직접 검증, fallback 이중 설계 적용
Agent 구조 학습 문서·블로그 수준 이해 OpenCode production 코드(compaction.ts 등) 직접 파싱 → 소스 수준 내재화
도구 계층 함수 호출 수준 MCP 프로토콜 기반 8개 도구 서버 — 어떤 AI 클라이언트든 연결 가능한 표준 구조
확장성 프레임워크 업데이트에 종속 직접 구현한 루프 → LangGraph StateGraph로 1:1 변환 가능, Multi-Agent로 자연스럽게 확장
"프레임워크 사용자"가 아니라,
프레임워크가 해결하는 문제를 먼저 직접 해결하고 — 그 위에서 프레임워크를 선택하는 개발자
현재 구현 → 프레임워크 연결점
지금 직접 구현한 것

tool_use → tool_result → 재호출 루프를 Spring Boot 코드로 직접 작성, 세션 관리·도구 실행·런타임 상태 조회를 포함한 커스텀 Agent Runtime

// round loop
while (stopReason == "tool_use") {
executeTools(toolUseBlocks);
response = callClaudeWithResults();
}
LangGraph로 변환 시

현재 루프의 각 단계가 LangGraph의 Node, 조건 분기가 Conditional Edge, 상태 전달이 StateGraph로 1:1 대응. 구조 이해가 완료되어 있어 변환 비용 최소

Planner Node
→ Generator Node
→ Validator Node (tool_use 결과 확인)
→ [오류 있음] Fixer Node → 재검증
→ [정상] END
* 이 포트폴리오의 핵심은 LangChain/LangGraph를 "사용"한 경험이 아니라, 그 프레임워크들이 추상화하는 문제를 먼저 직접 해결하고 내부 메커니즘을 이해한 경험입니다.
* HChat(사내 AI API) 기반 tool_use 검증 및 fallback 설계는 기업 환경 특유의 불확실성을 이미 경험한 사례로, 사내 Agentic AI 서비스 개발에 바로 적용 가능합니다.
