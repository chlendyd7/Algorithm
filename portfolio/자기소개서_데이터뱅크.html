S1. DB 병목 구조적 해결 (애기야가자)
성과/수치 문제해결 기술역량
적합한 질문: 직무역량, 성과 중심 자기소개, 어려웠던 점과 극복, 기술적 문제 해결 경험
200만 MAU 규모의 서비스에서 특정 인기 조회 API 응답 속도가 300ms 이상으로 상승하는 문제를 직면했습니다. 처음에는 인덱스 추가와 SQL 튜닝으로 대응했습니다. 일시적 개선은 있었지만, 트래픽이 몰리면 다시 지연이 발생했습니다. 문제의 본질은 쿼리가 아니라 '핫스팟 트래픽이 단일 DB에 집중되는 구조'였습니다. 접근 방식을 전면 수정했습니다. Redis 캐시 계층을 도입해 반복 조회 데이터를 분산하고, 조인 비용이 높은 테이블은 비정규화했으며, 수평적 파티셔닝을 적용해 읽기 트래픽을 분리했습니다. 결과: 평균 응답 속도 300ms → 180ms (40% 개선), 주요 쿼리 성능 30% 이상 향상, 메인 화면 로딩 1,500ms → 300ms. 이 경험으로 "성능 문제는 코드가 아니라 구조의 문제일 수 있다"는 것을 체감했습니다.

URL 경로와 필수 파라미터만 알면 로그인 없이도 계정 목록, 권한 목록, 설비 정보 등 민감 데이터를 조회할 수 있는 상태였습니다. 빠르게 처리하려면 특정 URL을 우회하거나 일부 엔드포인트만 막는 방법도 있었습니다. 그러나 그 방식은 공격자가 URL 패턴을 추측하면 다시 뚫릴 수 있는 미봉책이었습니다. 로그인 이후 데이터를 조회하는 모든 URL에 인증 및 세션 검증 로직을 추가하는 방식을 선택했습니다. 동일한 구조적 취약점이 다른 페이지에서도 반복되지 않도록 수평전개 기준을 세우고, 데이터 호출 구간 전체에 일관된 권한 검증 구조를 적용했습니다. 쉬운 수정이 있었지만 선택하지 않았습니다. 보안 문제는 빠른 처리보다 근본적인 구조 변경이 더 중요하다는 것을, 실제 보안 감사 결과를 마주하며 다시 확인했습니다.
취약한 접근 제어 (위험도: 상) 해소 전체 API 세션 검증 적용 수평전개 완료

S8. 실패 → 복구 → 구조 개선 (KEP 서버 장애)

현대위아 TMS MES 운영 중 KEP 서버(IO 서버) 보안 패치를 주말에 진행했습니다. 작업 완료로 판단했지만 현업에서 연락이 왔습니다. 실행해야 하는 프로그램을 전달받지 못해 누락했고, 현장 라인이 멈춘 것이었습니다. 소식을 듣고 1시간 일찍 출근해 직접 시스템을 정상화했습니다. 설비가 멈춘 것은 제 실수가 원인이었고, 빠른 복구 외에 선택지가 없었습니다. 복구 이후 두 가지를 바로 진행했습니다. 첫째, 패치 전 확인해야 할 프로그램 목록부터 재기동 절차까지 전체 작업 과정을 문서화했습니다. 다음 담당자가 같은 실수를 반복하지 않도록 만들기 위해서였습니다. 둘째, 더 근본적인 구조 문제를 해결했습니다. 기존 방식은 KEP 서버에 TAG를 직접 등록하는 방식이라, 변경이 생길 때마다 KEP 서버에 접근해야 했습니다. 다른 개발자와 논의하던 중 설정값을 내 DB에 저장하고 KEP 서버로 연결하는 방식을 알게 됐습니다. 유지보수성과 이식성이 명확히 높아지는 구조였기에 전환을 진행했습니다. 한 번의 실수가 문서화와 아키텍처 개선으로 이어진 경험이었습니다.
장애 발생 → 1시간 내 정상화 패치 절차 문서화 KEP 연동 구조 개선

현대위아 입사 후 사내에서 AI 코딩 도구 사용이 빠르게 늘고 있다는 것을 확인했습니다. 그러나 배포 프로세스, 보안 기준, 레거시 시스템 연결 방법 등 아무런 표준이 없다는 것도 동시에 보였습니다. 이 상태로 사용이 확산되면 보안 사고나 관리 불가 상태로 이어질 수 있다고 판단했습니다. 스스로 과제로 정하고 PoC 개발을 시작했습니다. 처음 접하는 기술들이 많았지만 직접 구현하면서 파악했습니다. Claude 기반 Agentic Loop를 구현하고, MCP 8개 도구를 통해 레거시 프로그램과 연동했습니다. CodeGenerator / EditAgent / SummaryAgent로 멀티 에이전트 구조를 분리하고, Agent 실행 추적(tracer)을 붙여 운영 가능한 수준의 구조를 만들었습니다. LangGraph, MCP, aiOps 개념도 구현하면서 직접 익혔습니다. 사내 AI 게이트웨이(HChat)와 연동해 기업 보안 정책 안에서 동작하도록 구성했습니다. 결과물을 입사 2달 만에 워크샵에서 발표했습니다. "사내 AI 코딩 도구를 어떻게 표준화할 것인가"라는 주제로, 현재 문제와 해결 방향, 구현 내용을 함께 제시했습니다. 모르는 기술도 직접 부딪히면 빠르게 흡수할 수 있다는 것을, 이 경험이 다시 확인해줬습니다.
입사 2달 만에 PoC 완성 워크샵 발표 MCP 8개 도구 멀티 에이전트 구조 사내 AI 게이트웨이 연동

S12. 불편함 → 직접 개선 — CI/CD 자동화 + K8s 배포 환경 구축 (CLICK-CEO)
