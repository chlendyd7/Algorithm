200만 MAU 규모의 서비스에서 특정 인기 조회 API 응답 속도가 300ms 이상으로 상승하는 문제를 직면했습니다. 처음에는 인덱스 추가와 SQL 튜닝으로 대응했습니다. 일시적 개선은 있었지만, 트래픽이 몰리면 다시 지연이 발생했습니다. 문제의 본질은 쿼리가 아니라 '핫스팟 트래픽이 단일 DB에 집중되는 구조'였습니다. 접근 방식을 전면 수정했습니다. Redis 캐시 계층을 도입해 반복 조회 데이터를 분산하고, 조인 비용이 높은 테이블은 비정규화했으며, 수평적 파티셔닝을 적용해 읽기 트래픽을 분리했습니다. 결과: 평균 응답 속도 300ms → 180ms (40% 개선), 주요 쿼리 성능 30% 이상 향상, 메인 화면 로딩 1,500ms → 300ms. 이 경험으로 "성능 문제는 코드가 아니라 구조의 문제일 수 있다"는 것을 체감했습니다.

S14. 동시성 제어 + 분산 트래픽 설계 — 연결 실패율 44% → 0% (CLICK-CEO)

적합한 질문: 대용량 트래픽 대응, 동시성/정합성 설계, 성능 개선, Kafka/Redis 활용, 결제 흐름 설계
Click-CEO는 호가창 기반 실시간 주식 시뮬레이션으로, 봇과 사용자가 동시에 대량의 주문을 생성하고 호가창이 실시간으로 갱신되는 구조였습니다. 폴링 방식을 WebSocket으로 전환한 직후 Stress test에서 연결 실패율이 44%까지 치솟았습니다. 단순히 연결 수를 늘리거나 서버 스펙을 높이는 방향이 아니라 원인부터 분석했습니다. 봇 주문 생성이 단일 스레드로 병목되며 WebSocket 연결 리소스를 고갈시키고 있었습니다. 처리 특성에 따라 저장소와 브로커를 분리했습니다. 호가창 매칭처럼 레이턴시가 중요한 실시간 핫패스는 Redis Sorted Set으로 처리했습니다. 주문 등록 시에는 DB 트랜잭션으로 잔고 확인과 예약 잠금을 원자적으로 처리해 이중 출금을 방지했고, 충분한 잔고가 확인된 주문만 호가창에 등록되도록 했습니다. 체결이 확정되면 DB 트랜잭션으로 매수자의 예약 잔고를 차감하고 매도자의 가용 잔고에 이전한 뒤, Kafka로 체결 이벤트를 발행했습니다. 체결 알림, 정산 집계, 봇 주문 배치처럼 순서 보장과 재처리가 중요한 로직은 Kafka Consumer가 독립적으로 소비하도록 분리해 체결 처리와 파생 작업의 결합도를 낮췄습니다. 이 구조로 연결 실패율이 0%로 해소됐습니다. 인프라 측면에서는 K8s HPA를 구성해 컨테이너 리소스 사용량이 70%를 초과하면 자동으로 파드를 추가 생성하도록 했습니다. Jenkins CasC와 K8s rolling update로 무중단 배포를 구현했고, 멀티 파드 환경에서 배치 작업이 중복 실행되는 문제는 ShedLock으로 제어했습니다.
