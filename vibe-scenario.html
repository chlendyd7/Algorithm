Agentic AI 확장 설계 · 업무 자동화 시나리오
현재 아키텍처의 다음 단계
VibeCoding Studio의 Agentic 구조를 기반으로 — 실제 업무 문제에 적용하면
현재 보유한 아키텍처 자산:
① tool_use 기반 자기검증 Agentic Loop (다회전 재호출, fallback 설계)
② MCP 서버 (어떤 AI 클라이언트든 연결 가능한 표준 도구 계층)
③ OpenCode 소스 분석으로 내재화한 Agent 역할 분리 패턴
④ FastAPI + Spring Boot + React 풀스택 + SSE 스트리밍 구현 경험

아래 시나리오는 이 자산을 "사내 업무 자동화"에 적용할 때의 설계 방향입니다.
지금 당장 해야 할 것 — 우선순위 순
🟢 즉시 (3일 이내)

LangGraph로 현재 루프 재구현

현재 Spring Boot에서 직접 구현한 tool_use → 재호출 루프를 LangGraph StateGraph로 변환. 코드 구조가 이미 완성되어 있어 Node/Edge로 1:1 매핑만 하면 됨.

→ "LangGraph 경험 있음"이 진짜 경험이 됨. 면접 때 "직접 만든 구조를 LangGraph로도 구현해봤다"는 가장 강력한 대답.
🟢 즉시 (2일 이내)

Streamlit으로 사내 업무 PoC 하나 완성

아래 시나리오 중 하나를 Streamlit UI + FastAPI + Claude API로 데모 수준 구현. UI는 Streamlit이라 하루면 됨. 실제로 돌아가는 데모가 있으면 포폴이 완전히 달라짐.

→ "Streamlit 경험 보유자" 우대 조건 충족 + 면접 때 데모 시연 가능.
🟡 단기 (1주)

Multi-Agent 실분리 구현

현재 단일 AIService를 CodeGeneratorAgent / SummaryAgent / EditAgent로 실제 분리. 역할별 모델 설정(Sonnet vs Haiku) 적용해 비용 최적화 측정.

→ "Multi-Agent 협업 구조 설계·개발" 직무 요구사항 직접 충족.
🟡 단기 (1주)

RAG 파이프라인 연동

기존 RAG 경험 기반으로 ChromaDB + 사내 문서 임베딩 파이프라인을 VibeCoding이나 별도 PoC에 통합. FastAPI 엔드포인트로 서빙.

→ RAG 경험을 현재 Agentic 구조와 결합해 "LLM API + RAG + FastAPI" 3개 우대 동시 충족.

Scenario 01 · Multi-Agent
주간 보고서 자동 초안 생성 Agent
데이터 조회 → 인사이트 도출 → 초안 작성 → 검토 흐름을 Agent 구조로 자동화
원인

주간 보고서 작성에 2~3시간 소모. 데이터 조회·정리·문장화·포맷팅을 매주 수동으로 반복. 담당자가 바뀌면 일관성도 깨짐

분석

VibeCoding에서 구현한 tool_use 루프와 동일한 패턴. 데이터 조회(fetch_kpi 도구) → 인사이트 생성 → 초안 작성 → 검토의 단계별 Agent 분리가 적합. OpenCode 분석에서 확인한 역할별 모델 최적화 적용 가능

설계 방향

DataFetchAgent(DB/API 조회, GPT-4o-mini) → AnalysisAgent(인사이트 도출, Claude Sonnet) → DraftAgent(초안 생성) → Streamlit 검토 UI. LangGraph StateGraph로 상태 전이 명시화

Scenario 02 · RAG + Agent
사내 문서 Q&A RAG Agent
내부 문서·매뉴얼·정책을 벡터 DB로 관리하고, AI가 질문에 맞는 문서를 찾아 답변 생성
원인

사내 문서가 분산되어 있어 필요한 정보를 찾는 데 시간 소모. 동일한 질문이 담당자에게 반복 접수. 문서 업데이트가 실시간으로 반영되지 않아 오래된 정보 참조 문제 발생

분석

정적 프롬프트에 문서를 전부 넣는 방식은 컨텍스트 길이 한계. 벡터 DB 기반 검색으로 질문과 유사한 문서만 선별 주입하는 RAG 구조 필요. VibeCoding의 런타임 상태 조회 패턴과 동일: "생성 전 필요한 정보를 먼저 조회"

설계 방향

문서 업로드 → ChromaDB 임베딩 파이프라인, 질문 시 Retriever가 상위 K개 청크 검색 → Claude API로 답변 생성. FastAPI로 서빙, Streamlit으로 Q&A 인터페이스. 출처 문서 링크 함께 반환

Scenario 03 · Agentic Workflow
업무 프로세스 자동화 — Single → Multi-Agent 전환
단일 LLM 호출로 처리하던 업무 자동화를 역할별 Agent로 분리해 품질·비용 동시 개선
원인

기존 LLM 활용이 "프롬프트 1회 → 결과 반환" 수준에 머물러, 복잡한 업무(다단계 판단, 외부 데이터 조회, 결과 검증)를 처리하지 못하고 오류율이 높음

분석

OpenCode 분석에서 확인한 패턴: 단일 Agent는 역할 혼재로 품질 저하. 역할 분리 시 각 Agent를 역할에 맞는 모델로 최적화 가능. VibeCoding의 tool_use 루프가 이 구조의 토대

설계 방향

Planner(작업 분해) → Executor(도구 실행 + 자기검증 루프) → Reviewer(결과 품질 확인) → SummaryAgent(Haiku, 저비용). LangGraph StateGraph로 조건 분기 명시화. fallback은 VibeCoding 구현 방식 그대로 적용

현재 구현 → 시나리오 연결점

이미 해결한 문제가 모든 시나리오의 토대입니다

1
tool_use 루프 → 모든 시나리오의 "도구 호출 + 결과 반영" 패턴 동일. 구조를 이미 이해하고 구현했기 때문에 업무 도메인만 바꾸면 적용 가능
2
fallback 설계 → 기업 환경에서 API 제약·장애 상황에 graceful degradation 적용. 모든 PoC에서 즉시 재사용 가능
3
MCP 서버 → 시나리오에서 "도구 실행 계층"으로 그대로 활용. fetch_kpi, search_docs 같은 도구를 추가만 하면 됨
4
OpenCode 분석 패턴 → Multi-Agent 역할 분리·모델 최적화·Compaction을 이미 설계 완료. LangGraph로 코드화하는 것만 남음
* 위 시나리오는 현재 VibeCoding Studio에서 구현한 Agentic 아키텍처를 기반으로 한 설계 방향입니다.
* 각 시나리오의 핵심 패턴(tool-calling 루프, fallback, Multi-Agent 역할 분리)은 이미 구현 완료된 구조이며, 업무 도메인과 도구 정의만 변경하면 적용 가능합니다.
* LangGraph StateGraph 재구현 및 Streamlit PoC는 현재 진행 예정입니다.
