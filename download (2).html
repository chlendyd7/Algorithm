React·Spring·FastAPI 기반의 Bolt 스타일 바이브코딩 웹앱 서비스를 개발하며, AI가 생성 중 필요한 런타임 정보를 직접 조회하고 결과를 반영하는 Agentic Loop를 설계·구현했습니다. 이후 이를 LangChain/LangGraph 관점에서 재해석하며, 제가 먼저 해결한 문제가 곧 tool-calling 표준화와 상태 기반 orchestration의 문제였음을 정리했습니다.

핵심 포지셔닝
  • 단순 LLM API 호출 기능이 아니라, AI가 도구를 호출하고 결과를 반영하며 스스로 판단을 이어가는 Agentic AI 실행 구조를 직접 설계·구현
  • 운영자용 MCP 서버(8개 도구)와 코드생성용 인라인 tool 호출을 별도 기능이 아닌 공통 tool-calling 인프라로 바라보며 아키텍처 일관성 확보
  • LangChain/LangGraph를 먼저 사용한 것이 아니라, 프레임워크가 해결하는 문제를 먼저 커스텀하게 해결한 뒤 이를 표준화·상태화하는 방향으로 정리
“프레임워크 사용자”가 아니라, 프레임워크가 정리하는 문제를 먼저 직접 해결한 백엔드 개발자
역할
  • React·Spring·FastAPI 기반 바이브코딩 웹앱 서비스를 개인 프로젝트로 설계·구현하며, AI 코드생성 기능의 백엔드 아키텍처 전반을 주도
  • 사용자 프롬프트를 입력받아 생성 흐름을 시작하는 코드생성 요청 API와, 반복 수정 요청의 문맥을 유지하는 대화 세션 관리 구조를 구현
  • 모델의 tool_use → tool_result → 재호출 → 종료 흐름을 직접 설계해, 생성 과정에서 프로젝트 상태를 조회하고 반영하는 자기검증형 Agentic Loop 구현
  • 운영자용 MCP 서버(8개 도구)를 설계하여 배포/버전/상태조회/앱 데이터 관리 등으로 확장 가능한 도구 계층 기반 마련
  • 현재 Single Agent 중심 구조를 기반으로, 향후 코드생성·검증·요약·운영지원 역할 분리형 Multi-Agent workflow로 확장 가능한 구조 설계
주요성과
  • 단발성 1회 프롬프트 호출 구조의 한계로 발생하던 API 경로·라우팅·import 오류 가능성을 줄이기 위해, AI가 생성 도중 현재 프로젝트 상태를 직접 조회하는 tool-calling 기반 자기검증 루프를 구현
  • 운영자용 MCP와 코드생성용 인라인 tool 호출을 공통 tool-calling 인프라로 구조화하여, 사용자 지원과 AI 자기검증을 함께 지원하는 재사용 가능한 Agentic AI 아키텍처 관점 확보
  • 실제 로그에서 round=0 tool_use → list_vibe_apis 호출 → round=1 end_turn 흐름을 검증하여, HChat Claude API의 tool_use 지원 여부와 적용 가능성을 직접 확인
  • 외부 API의 tool_use 지원 불확실성을 고려해 기존 생성 방식으로 자동 전환되는 fallback 구조를 설계하여, PoC 단계에서도 안정적 실행 흐름 확보
  • 현재 커스텀 루프를 LangChain의 구성요소 모듈화, LangGraph의 상태 기반 orchestration 관점으로 정리하여, 향후 표준 프레임워크 기반 고도화 방향 수립

프레임워크로 정립 가능한 것

문제 해결 방식
Problem
정적 프롬프트만으로는 생성 시점의 실제 프로젝트 상태를 반영하기 어려워, 이미 존재하는 API와 충돌하거나 잘못된 경로를 참조하는 코드가 반복 생성되는 문제가 있었음
Approach
FastAPI 기반 백엔드에서 코드생성 요청, 세션 관리, tool 실행 중계를 담당하고, AI가 생성 도중 필요한 런타임 정보를 도구로 직접 조회하도록 설계하여 자기검증형 Agentic Workflow 구현
Result
실제 `tool_use` 로그 검증을 통해 적용 가능성을 확인했고, 단순 생성 기능이 아닌 서비스형 Agentic AI 실행 구조와 공통 tool-calling 아키텍처 관점을 확보
Next Step
현재 커스텀 루프를 LangChain/LangGraph 기반으로 모듈화·상태화하고, 코드생성·검증·요약·운영지원 역할 분리형 Multi-Agent workflow로 확장 예정
* 이 프로젝트의 핵심은 LangChain/LangGraph를 먼저 사용한 것이 아니라, 그 프레임워크들이 해결하려는 문제를 이미 커스텀 agent runtime으로 직접 해결했다는 점입니다.
* 따라서 본 프로젝트는 단순 생성형 AI 기능 구현이 아니라, 서비스형 Agentic AI 아키텍처를 설계하고 이를 프레임워크 관점으로 체계화한 경험으로 설명할 수 있습니다.
